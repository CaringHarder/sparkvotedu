<objective>
Build the student-facing voting interface with two viewing modes: Simple (one matchup at a time for younger students) and Advanced (full bracket view for older students). Both modes use optimistic vote feedback and support real-time bracket updates.

Purpose: This is the student's primary interaction with the bracket -- the moment of engagement where they cast votes. The simple/advanced split ensures the UI is age-appropriate. Optimistic updates make voting feel instant even on slower connections.
Output: Student bracket voting page with simple and advanced views, matchup vote card component.
</objective>

  <action>
    **1. Create src/components/bracket/matchup-vote-card.tsx** (`'use client'`):

    A reusable card component for voting on a single matchup. Used by both simple and advanced views.

    Props: `{ matchup: MatchupData; participantId: string; initialVote: string | null; disabled?: boolean; showResult?: boolean }`

    Structure:
    - Uses `useVote(matchup.id, participantId, initialVote)` hook for optimistic voting
    - Renders two large clickable entrant buttons side by side (or stacked on mobile)
    - Each button shows the entrant name in large text
    - On click: calls `submitVote(entrant.id)` from the hook
    - Voted entrant: highlighted with primary color border/background + checkmark icon
    - `disabled` prop: when matchup status is not "voting" (pending or decided), buttons are non-interactive
    - `showResult` prop: when true and matchup is decided, shows the winner with a crown/trophy icon
    - If `isPending` from hook: show a subtle loading indicator on the voted button
    - If `error` from hook: show a brief error toast/message that auto-dismisses
    - "VS" text between the two entrant buttons
    - Matchup status indicator: small badge showing "Open" (voting), "Waiting" (pending), "Decided" (decided)
    - Tailwind styling: rounded-xl cards, hover:scale-105 on interactive buttons, transition-all for smooth feedback
    - Use `React.memo` on the component to prevent unnecessary re-renders when parent updates

    **2. Create src/components/student/simple-voting-view.tsx** (`'use client'`):

    Simple mode for younger students. Shows one matchup at a time with sequential navigation.

    Props: `{ matchups: MatchupData[]; participantId: string; bracketName: string; initialVotes: Record<string, string | null> }`

    Structure:
    - Filters matchups to current round (matchups where status is "voting")
    - Maintains `currentIndex` state to track which matchup is shown
    - Shows a single `MatchupVoteCard` for the current matchup, large and centered
    - Navigation: "Previous" and "Next" buttons below the card (disabled at bounds)
    - Progress indicator: "Matchup 2 of 4" text
    - If no matchups are open for voting: show a friendly "Waiting for your teacher to open voting..." message with a subtle pulse animation
    - After voting on all matchups: show "All done! Waiting for results..." with a checkmark animation
    - Subscribes to `useRealtimeBracket` for structural updates (round changes open new matchups)
    - When new matchups become votable (bracket_update event), auto-navigates to the first new matchup
    - Uses `useTransportFallback` for WebSocket detection -- if polling mode, manually refetch matchup state on interval
    - Minimal chrome: no bracket diagram, no header navigation beyond bracket name
    - Large fonts and big tap targets for younger students (min-h-16 on buttons)

    **3. Create src/components/student/advanced-voting-view.tsx** (`'use client'`):

    Advanced mode for older students. Shows the full bracket with all matchups visible and votable.

    Props: `{ bracket: BracketWithDetails; participantId: string; initialVotes: Record<string, string | null> }`

    Structure:
    - Renders the existing `BracketDiagram` component (from Phase 3) as the main view
    - Extends/wraps BracketDiagram to make votable matchup boxes interactive (clickable)
    - Alternatively, creates a new `BracketDiagramInteractive` that wraps BracketDiagram with clickable overlays on votable matchups
    - When a votable matchup is clicked: opens a modal/overlay with `MatchupVoteCard` for that matchup
    - Subscribes to `useRealtimeBracket(bracket.id)` for live updates
    - When bracket updates (winner selected, round advanced): refetch bracket state and update the diagram
    - Voted matchups show a small checkmark badge on the bracket diagram
    - Uses `useTransportFallback` for WebSocket detection
    - Shows bracket name and current round at the top
    - Responsive: on mobile, the bracket diagram may need horizontal scrolling (existing behavior from Phase 3)

    **Key UX decisions to implement:**
    - Vote counts are NEVER shown to students (CONTEXT.md: "hidden until teacher reveals")
    - Vote changes ARE allowed (RESEARCH recommendation: allow changes until matchup closes). Tapping a different entrant calls `submitVote` again, which upserts via the server action.
    - No confirmation step: single tap/click to cast vote (CONTEXT.md: "classroom speed priority")
    - After voting: entrant highlights with checkmark (CONTEXT.md decision)
  </action>

    **1. Create src/app/(student)/session/[sessionId]/bracket/[bracketId]/page.tsx:**

    This is a Server Component that fetches bracket data and routes to the appropriate voting view (simple or advanced).

    Structure:
    - Extract `sessionId` and `bracketId` from `params`
    - Fetch the bracket with details: `prisma.bracket.findUnique({ where: { id: bracketId }, include: { entrants: { orderBy: { seedPosition: 'asc' } }, matchups: { include: { entrant1: true, entrant2: true, winner: true }, orderBy: [{ round: 'asc' }, { position: 'asc' }] } } })`
    - Validate bracket exists and belongs to the session (bracket.sessionId === sessionId). If not, show `notFound()`.
    - Validate bracket status is "active". If draft, show "This bracket isn't active yet." If completed, show the final bracket result.
    - Read participant identity from cookies or params. The student identity comes from the client-side `localStorage` (pattern from Phase 2). Since this is a Server Component, we cannot access localStorage directly. Instead:
      - Accept `participantId` as a search param (`?pid=xxx`) -- the student session layout or navigation passes this
      - OR: Create a thin client wrapper component that reads localStorage and passes participantId down
      - Recommended: Use a client component wrapper that reads `localStorage` key `sparkvotedu_session_${sessionId}` (established pattern from Phase 2), extracts `participantId`, and renders the appropriate view
    - Fetch initial votes for this participant: for each matchup, check if participant has voted using `hasVoted` from vote DAL (or batch query: `prisma.vote.findMany({ where: { participantId, matchupId: { in: matchupIds } }, select: { matchupId: true, entrantId: true } })`)
    - Build `initialVotes` map: `Record<matchupId, entrantId | null>`
    - Based on `bracket.viewingMode`:
