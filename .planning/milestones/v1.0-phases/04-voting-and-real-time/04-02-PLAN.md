<feature>
  <name>Bracket Advancement Engine</name>
  <files>src/lib/bracket/advancement.ts, src/lib/bracket/__tests__/advancement.test.ts</files>
  <behavior>
    The advancement engine handles winner propagation through tournament rounds:

    **advanceMatchupWinner(matchupId, winnerId, bracketId):**
    - Sets winnerId on the matchup
    - Sets matchup status to "decided"
    - If matchup has nextMatchupId, propagates winner to the next matchup
    - Slot assignment: odd positions (1, 3, 5, 7) -> entrant1Id, even positions (2, 4, 6, 8) -> entrant2Id
    - All within a Prisma $transaction for atomicity
    - Returns the updated matchup

    **undoMatchupAdvancement(matchupId, bracketId):**
    - Validates matchup has a winner (error if no winner)
    - Checks if next matchup has any votes (error if votes exist -- cannot undo once next round has voting)
    - Clears winnerId on the matchup, sets status back to "voting"
    - Clears the propagated entrant from the next matchup (entrant1Id or entrant2Id based on position parity)
    - All within a Prisma $transaction

    **batchAdvanceRound(bracketId, round):**
    - Finds all matchups in the given round that have a winnerId but next matchup not yet populated
    - Propagates all winners to their respective next matchups
    - Returns count of matchups advanced

    **checkRoundComplete(bracketId, round):**
    - Returns true if all matchups in the given round have status "decided"
    - Returns false otherwise

    **isBracketComplete(bracketId):**
    - Returns true if the final matchup (highest round, position 1) has a winnerId
    - Returns the winnerId if complete, null otherwise

    Test cases:
    - 4-entrant bracket: advance R1M1 winner -> appears as entrant1 of R2M1 (position 1 is odd -> entrant1)
    - 4-entrant bracket: advance R1M2 winner -> appears as entrant2 of R2M1 (position 2 is even -> entrant2)
    - 8-entrant bracket: full advancement through 3 rounds
    - Undo: advancing then undoing restores previous state
    - Undo blocked: next matchup with votes returns error
    - Round complete check: all decided vs some pending
    - Bracket complete: final matchup has winner
  </behavior>
  <implementation>
    Use Prisma interactive transactions (`prisma.$transaction(async (tx) => { ... })`) for all mutations. The slot assignment logic (odd -> entrant1, even -> entrant2) matches the bracket engine's `Math.ceil(position / 2)` next matchup calculation from Phase 3. For the undo vote check, count votes on the next matchup -- if > 0, block the undo with a descriptive error message. Import `prisma` from `@/lib/prisma`.

    For testing: Since this involves database operations, tests should use `vi.mock` to mock the Prisma client, testing the logic of slot assignment, status transitions, and error conditions without requiring a real database. Create mock matchup data that mirrors the bracket engine's output structure. Alternatively, if the existing test setup supports database tests, use real transactions with rollback.

    Follow the existing pattern from `src/lib/bracket/__tests__/engine.test.ts` for test structure (describe blocks, clear test names).
  </implementation>
</feature>
