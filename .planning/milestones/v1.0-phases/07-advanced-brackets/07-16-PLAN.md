<task type="auto">
  <name>Task 1: Fix large bracket creation timeout</name>
  <files>src/lib/dal/bracket.ts, src/actions/bracket.ts</files>
  <action>
  **1a. Add transaction timeout to createBracketDAL** (src/lib/dal/bracket.ts):

  Find the `prisma.$transaction(async (tx) => {` call in createBracketDAL (line 223). Change it to:
  ```
  const bracket = await prisma.$transaction(async (tx) => {
    // ... existing transaction body unchanged ...
  }, { timeout: 30000 })
  ```

  The second argument `{ timeout: 30000 }` overrides Prisma's default 5s interactive transaction timeout to 30 seconds. This is sufficient for size 64 (~190 ops at ~50ms each = ~9.5s worst case).

  **1b. Add transaction timeout to createDoubleElimBracketDAL** (same file):

  Find the `prisma.$transaction(async (tx) => {` call in createDoubleElimBracketDAL (around line 346). Add the same timeout:
  ```
  const bracket = await prisma.$transaction(async (tx) => {
    // ... existing transaction body unchanged ...
  }, { timeout: 30000 })
  ```

  DE brackets with 64 entrants generate even more matchups (WB + LB + GF), so timeout is critical here too.

  **1c. Improve error logging in createBracket action** (src/actions/bracket.ts):

  Find the catch block at lines 103-105:
  ```
  } catch {
    return { error: 'Failed to create bracket' }
  }
  ```

  Change to:
  ```
  } catch (err) {
    console.error('createBracket failed:', err)
    return { error: 'Failed to create bracket' }
  }
  ```

  This ensures the actual Prisma timeout error (or any other error) is logged to the server console for debugging, instead of being silently swallowed.

  **Do NOT refactor sequential creates to createMany in this plan.** The timeout override is the minimal fix. Batching optimization can be done in a future performance pass. The goal is to unblock UAT test 16 with minimal risk.
