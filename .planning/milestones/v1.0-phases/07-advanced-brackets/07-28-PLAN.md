Two fixes in live-dashboard.tsx:

**Fix 1: Fallback celebration race condition (lines ~308-313)**

The fallback celebration effect fires at T+2s before the chained celebration at T+5s because `hasShownRevealRef` is not checked inside the setTimeout callback. The ref IS checked in the effect condition, but by the time the setTimeout fires, the ref may not have been set yet when the effect was evaluated.

Change the fallback celebration effect from:
```tsx
useEffect(() => {
  if (bracketCompleted && !revealState && !hasShownRevealRef.current) {
    const timer = setTimeout(() => setShowCelebration(true), 2000)
    return () => clearTimeout(timer)
  }
}, [bracketCompleted, revealState])
```

To add a double-check INSIDE the setTimeout callback:
```tsx
useEffect(() => {
  if (bracketCompleted && !revealState && !hasShownRevealRef.current) {
    const timer = setTimeout(() => {
      // Double-check: if reveal path already handled celebration, skip fallback
      if (!hasShownRevealRef.current) {
        setShowCelebration(true)
      }
    }, 2000)
    return () => clearTimeout(timer)
  }
}, [bracketCompleted, revealState])
```

This ensures the fallback only fires if the primary reveal path hasn't already claimed the celebration. The ref check at T+2s will see `hasShownRevealRef.current === true` (set by the status-transition effect) and skip the duplicate.
