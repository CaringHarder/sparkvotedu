## Summary

Phase 7.1 adds a third resolution mode ("auto") to the existing predictive bracket system. In this mode, student predictions are automatically tabulated as votes to determine matchup winners, eliminating the voting rounds that the "manual" and "vote_based" modes require. The teacher controls a staged reveal flow: close predictions, prepare (tabulate) results, optionally override winners, then release results round-by-round to students.

The existing codebase provides a solid foundation: the `Prediction` model, `submitPredictions` DAL, `scorePredictions` engine, `usePredictions` hook, `PredictiveBracket` component, and `PredictionLeaderboard` are all implemented and working. The `predictiveResolutionMode` field on the Bracket model already stores `'manual' | 'vote_based'` -- it needs to accept `'auto'` as a third value. The primary new work is: (1) a tabulation engine that converts predictions into matchup winners, (2) a teacher preview/override UI with round-by-round tie-breaking, (3) a teacher-paced reveal flow with "Reveal Next Round" controls, (4) a student reveal experience with progressive bracket fill-in, color-coded accuracy, and animated leaderboard, and (5) a top-3 podium celebration replacing the existing champion celebration.

The architecture follows the exact same patterns used by the existing bracket types: server actions for mutations, DAL functions for data access, Supabase Broadcast for real-time updates, and `motion/react` for animations. No new dependencies are needed.

**Primary recommendation:** Extend the existing `PredictiveResolutionMode` type to include `'auto'`, add new prediction status values for the staged flow (`'tabulating'`, `'previewing'`, `'revealing'`), and build the tabulation engine as a pure function alongside `scorePredictions` in `src/lib/bracket/predictive.ts`. Build the reveal experience as new components that compose existing building blocks (BracketDiagram, PredictionLeaderboard, WinnerReveal countdown, PresentationMode wrapper).

### Anti-Patterns to Avoid
- **Resolving all rounds simultaneously during tabulation:** Tabulation must be round-by-round because later-round matchup entrants depend on earlier-round outcomes. Calculate Round 1 winners first, propagate to Round 2 matchups, then calculate Round 2 winners, etc.
- **Showing prediction vote counts to students:** The decisions explicitly say "prediction vote counts are never shown to students -- only the outcome matters." Only the teacher preview shows winners (no percentages).
- **Auto-advancing after reveal:** Teacher must manually click "Reveal Next Round" for each round. Do NOT auto-advance on a timer. The countdown is just a visual effect between rounds.
- **Modifying the existing manual/vote_based flows:** Phase 7.1 adds a third mode. The existing two modes must remain untouched.
- **Creating a separate bracket model for auto-resolved brackets:** Use the existing Bracket + Matchup models with the `predictiveResolutionMode: 'auto'` discriminator. Add new predictionStatus values to the same field.

## Common Pitfalls

### Pitfall 1: Round-by-Round Tabulation Cascade Errors
**What goes wrong:** Later-round winners are calculated based on Round 1 predictions, not Round 1 outcomes. Student A predicted X wins Round 1, but the majority picked Y. The tabulation uses X as an entrant in Round 2 instead of Y.
**Why it happens:** Treating all predictions as independent instead of propagating tabulated winners forward.
**How to avoid:** Tabulate strictly in round order. After resolving Round N, propagate winners to Round N+1 matchup slots. Only then tabulate Round N+1 predictions. For Round N+1, only count predictions that match the actual Round N+1 entrants (ignore predictions for entrants eliminated in earlier rounds).
**Warning signs:** Students who predicted the Round 1 loser appear to have "predictions" for later rounds that reference eliminated entrants.

### Pitfall 2: Teacher Override Cascade Invalidation
**What goes wrong:** Teacher overrides a Round 1 winner, but Round 2+ auto-resolved results still reflect the original tabulation.
**Why it happens:** Overrides don't cascade through the bracket structure.
**How to avoid:** When a teacher overrides a matchup in the preview, re-tabulate all downstream rounds. The preview UI should show a "Recalculating..." state and update all affected matchups. This is similar to the existing `usePredictionCascade` hook that clears downstream predictions when a student changes an earlier pick.
**Warning signs:** After overriding Round 1, Round 2 still shows the original auto-resolved winners.

### Pitfall 3: Concurrent Prediction Close + Tabulation Race
**What goes wrong:** Teacher closes predictions and immediately clicks "Prepare Results" before all prediction submissions have been written to the database.
**Why it happens:** The prediction close broadcast may reach some students before their in-flight submissions complete.
**How to avoid:** Add a brief delay or explicit "predictions are closed" confirmation state between closing predictions and allowing tabulation. The `predictionStatus: 'tabulating'` transition should only be allowed from `predictions_open` after a minimum 2-second buffer. Alternatively, the tabulation action should use a database snapshot that includes all predictions written before the close timestamp.
**Warning signs:** Tabulation produces different results than a manual count of predictions in the database.

### Pitfall 4: Empty Later-Round Predictions
**What goes wrong:** Many students only predict Round 1, leaving later rounds empty. The "no predictions" status triggers for every later-round matchup, forcing the teacher to manually resolve most of the bracket.
**Why it happens:** Simple mode predictions don't cascade like advanced mode. Students may not realize they need to predict all rounds.
**How to avoid:** The existing `usePredictionCascade` hook already propagates predictions through rounds in advanced mode, and the simple mode requires all matchups to be selected before submit (enforced by `allSelected` check). Both modes should produce complete brackets. However, some students may submit partial predictions (before the `allSelected` gate was enforced, or via API). The tabulation should gracefully handle sparse predictions: if a matchup has > 0 total predictions, use majority; if exactly 0, flag as 'no_predictions' for teacher.
**Warning signs:** Tabulation returns many 'no_predictions' matchups despite many students having submitted predictions.

### Pitfall 5: Reveal State Desync Between Teacher and Students
**What goes wrong:** Teacher reveals Round 2, but some students still show Round 1 as the latest revealed round.
**Why it happens:** Supabase broadcast may be lost or delayed on school networks. Polling fallback may not have fired yet.
**How to avoid:** The student reveal view should always fetch current state from the server on broadcast receipt (which is the existing pattern in `useRealtimeBracket`). Include the `revealedUpToRound` in the bracket state API response so students can reconcile. Also: the matchup statuses in the database are the source of truth -- if a matchup has `status: 'decided'`, it has been revealed regardless of broadcast delivery.
**Warning signs:** Students see stale bracket state that doesn't match what the teacher sees.

### Pitfall 6: Prediction Status vs Bracket Status Confusion
**What goes wrong:** New prediction statuses (tabulating, previewing, revealing) conflict with existing bracket status (draft, active, completed).
**Why it happens:** The existing system already has some complexity: `bracket.status` tracks the overall lifecycle, while `bracket.predictionStatus` tracks the prediction-specific lifecycle. Adding more values to `predictionStatus` could create ambiguous states.
**How to avoid:** Keep `bracket.status` simple (stays 'active' throughout the entire auto-resolution flow, transitions to 'completed' only at the end). All new state lives in `predictionStatus`. The bracket status transition is: `draft -> active (when predictions open) -> completed (when teacher clicks Complete Bracket after reveal)`. The prediction status transition is: `predictions_open -> tabulating -> previewing -> revealing -> completed`.
**Warning signs:** Bracket shows as "completed" in the teacher's bracket list before the reveal is done.

### Pitfall 7: Leaderboard Top-10 Plus Self-Rank Logic
**What goes wrong:** The leaderboard shows all students instead of top 10 + self. Or the student's own rank is shown as "11" when they're actually tied for 10th.
**Why it happens:** Naive implementation just slices the array to 10 without checking if the current student is included.
**How to avoid:** Calculate full leaderboard, find the current student's position. If position <= 10, return top 10. If position > 10, return top 10 plus the student's row with their actual rank. Handle ties: if multiple students have the same score, they share the same rank, and the next rank skips accordingly.
**Warning signs:** Student sees their rank as undefined or missing from the leaderboard.

### Prediction Tabulation Engine
```typescript
// Source: New pure function, following scorePredictions pattern in predictive.ts

interface TabulationInput {
  matchupId: string
  round: number
  position: number
  entrant1Id: string
  entrant2Id: string
  isBye: boolean
  nextMatchupId: string | null
}

interface TabulationResult {
  matchupId: string
  round: number
  position: number
  winnerId: string | null
  entrant1Id: string
  entrant2Id: string
  entrant1Votes: number
  entrant2Votes: number
  totalVotes: number
  status: 'resolved' | 'tie' | 'no_predictions'
}

/**
 * Tabulate predictions as votes round-by-round, cascading winners forward.
 *
 * For each round (ascending):
 * 1. For each matchup in the round, count predictions per entrant
 * 2. Majority wins -> status: 'resolved'
 * 3. Exact tie -> status: 'tie', winnerId: null (teacher breaks)
 * 4. Zero predictions -> status: 'no_predictions', winnerId: null (teacher decides)
 * 5. Propagate resolved winners to next round matchups
 *
 * Returns results for ALL matchups (including byes which are pre-resolved).
 */
function tabulatePredictions(
  predictions: Array<{ matchupId: string; predictedWinnerId: string }>,
  matchups: TabulationInput[],
  totalRounds: number
): TabulationResult[] {
  // Build prediction lookup: matchupId -> { entrantId -> count }
  const predsByMatchup = new Map<string, Map<string, number>>()
  for (const pred of predictions) {
    if (!predsByMatchup.has(pred.matchupId)) {
      predsByMatchup.set(pred.matchupId, new Map())
    }
    const counts = predsByMatchup.get(pred.matchupId)!
    counts.set(pred.predictedWinnerId, (counts.get(pred.predictedWinnerId) ?? 0) + 1)
  }

  // Build matchup lookup and track which matchup feeds which
  const matchupById = new Map(matchups.map(m => [m.matchupId, m]))
  const results: TabulationResult[] = []

  // Track resolved winners for cascade
  const resolvedWinners = new Map<string, string>() // matchupId -> winnerId

  for (let round = 1; round <= totalRounds; round++) {
    const roundMatchups = matchups
      .filter(m => m.round === round && !m.isBye)
      .sort((a, b) => a.position - b.position)

    for (const matchup of roundMatchups) {
      const counts = predsByMatchup.get(matchup.matchupId) ?? new Map()

      // Only count predictions for the actual entrants in this matchup
      const e1Votes = counts.get(matchup.entrant1Id) ?? 0
      const e2Votes = counts.get(matchup.entrant2Id) ?? 0
      const totalVotes = e1Votes + e2Votes

      let winnerId: string | null = null
      let status: 'resolved' | 'tie' | 'no_predictions'

      if (totalVotes === 0) {
        status = 'no_predictions'
      } else if (e1Votes > e2Votes) {
        winnerId = matchup.entrant1Id
        status = 'resolved'
      } else if (e2Votes > e1Votes) {
        winnerId = matchup.entrant2Id
        status = 'resolved'
      } else {
        status = 'tie'
      }

      results.push({
        matchupId: matchup.matchupId,
        round: matchup.round,
        position: matchup.position,
        winnerId,
        entrant1Id: matchup.entrant1Id,
        entrant2Id: matchup.entrant2Id,
        entrant1Votes: e1Votes,
        entrant2Votes: e2Votes,
        totalVotes,
        status,
      })

      if (winnerId) {
        resolvedWinners.set(matchup.matchupId, winnerId)
      }
    }

    // Propagate resolved winners to next round matchups
    // This updates the matchups array in-place for the next iteration
    for (const matchup of roundMatchups) {
      const winnerId = resolvedWinners.get(matchup.matchupId)
      if (!winnerId || !matchup.nextMatchupId) continue

      const nextMatchup = matchupById.get(matchup.nextMatchupId)
      if (!nextMatchup) continue

      // Slot assignment: odd position -> entrant1, even -> entrant2
      if (matchup.position % 2 === 1) {
        nextMatchup.entrant1Id = winnerId
      } else {
        nextMatchup.entrant2Id = winnerId
      }
    }
  }

  return results
}
```

### Reveal Round Server Action Pattern
```typescript
// Source: Follows advanceMatchup action pattern in bracket-advance.ts

// In actions/prediction.ts:
export async function revealNextRound(input: unknown) {
  const teacher = await getAuthenticatedTeacher()
  if (!teacher) return { error: 'Not authenticated' }

  const parsed = z.object({
    bracketId: z.string().uuid(),
    round: z.number().int().min(1),
  }).safeParse(input)
  if (!parsed.success) return { error: 'Invalid data' }

  const { bracketId, round } = parsed.data

  // Verify bracket ownership + correct status
  const bracket = await prisma.bracket.findFirst({
    where: { id: bracketId, teacherId: teacher.id, predictionStatus: 'revealing' },
    select: { id: true, sessionId: true },
  })
  if (!bracket) return { error: 'Bracket not found or not in reveal state' }

  // Fetch round matchups that should be revealed
  // (these already have winnerId set during tabulation/preview phase)
  const roundMatchups = await prisma.matchup.findMany({
    where: { bracketId, round, status: 'pending' },
    select: { id: true, winnerId: true },
  })

  if (roundMatchups.length === 0) return { error: 'No matchups to reveal for this round' }

  // Verify all matchups have winners (teacher resolved all ties in preview)
  const unresolved = roundMatchups.filter(m => !m.winnerId)
  if (unresolved.length > 0) return { error: `${unresolved.length} matchups still need winners` }

  // Mark all round matchups as 'decided'
  await prisma.matchup.updateMany({
    where: { bracketId, round, status: 'pending' },
    data: { status: 'decided' },
  })

  // Broadcast reveal event
  await broadcastBracketUpdate(bracketId, 'reveal_round', { round })

  revalidatePath(`/brackets/${bracketId}`)

  return { success: true, round }
}
```

## Discretionary Recommendations

### 1. Students Can Revisit Completed Bracket and Leaderboard After Session
**Recommendation:** Follow existing bracket completion patterns -- the completed bracket is viewable read-only with the final leaderboard. The student bracket voting page already handles `status === 'completed'` state and renders a read-only view. Add the leaderboard tab to the completed view for auto-resolved brackets.
**Rationale:** Consistency with existing behavior. Students may want to review their predictions and compare with outcomes after the session ends.

### 2. Countdown Animation Between Rounds
**Recommendation:** Reuse the `WinnerReveal` component's countdown pattern (3-2-1 with `motion/react`), but as a standalone overlay that does NOT auto-dismiss to celebration. Show "Round X Results..." text after countdown, then reveal. The countdown only appears between rounds (after teacher clicks "Reveal Next Round"), not for Round 1.
**Rationale:** The existing WinnerReveal already implements exactly this animation with reduced motion support. Extracting the countdown into a standalone component avoids duplicating animation logic.

### 3. Presentation Mode Implementation
**Recommendation:** Reuse the `PresentationMode` component from `src/components/poll/presentation-mode.tsx` directly. Add "F" key shortcut to the teacher's reveal dashboard. Let teacher toggle between bracket and leaderboard views within presentation mode using tab buttons.
**Rationale:** The poll presentation mode already implements Fullscreen API, dark theme, exit handling, and the F key shortcut. Zero new code needed for the container -- only the content changes.

### 4. Leaderboard Rank Animation
**Recommendation:** Use `motion/react`'s `layout` animation prop on leaderboard rows. When scores change after a round reveal, rows with `layout` prop will automatically animate to their new positions. Use `LayoutGroup` to scope the animations. Spring physics (`type: 'spring'`, `stiffness: 300`, `damping: 30`) give a natural feel.
**Rationale:** `motion/react` is already in the project and its layout animation system is the standard approach for list reordering. No custom animation logic needed -- just add the `layout` prop to existing `<tr>` or `<div>` elements.

### 5. "Prepare Results" Loading Communication
**Recommendation:** When teacher clicks "Prepare Results", show an inline loading state: "Tabulating predictions..." with a spinner, transitioning to "Results ready" with a check mark. Use the existing `useTransition` pattern (already used throughout the codebase for pending states). The tabulation itself is fast (pure function + a few DB writes), so a progress bar is unnecessary.
**Rationale:** The tabulation will take < 1 second for any realistic bracket size (30 students x 63 matchups = 1,890 predictions). A simple spinner + success message follows the established patterns in the codebase.

## Open Questions

1. **Storing tabulation results during preview phase**
   - What we know: Tabulation produces a `TabulationResult[]` with winners for each matchup. These need to persist so the teacher can review/override before release.
   - What's unclear: Whether to store tabulated winners directly as `winnerId` on Matchup records during preview (before reveal), or in a temporary structure.
   - Recommendation: Store directly on Matchup records (set `winnerId` but keep `status: 'pending'`). This is simpler and follows the existing pattern where `winnerId` is set before status changes to 'decided'. The reveal action only changes status from 'pending' to 'decided'. Override changes the `winnerId` directly. Rollback (reopen predictions) clears all `winnerId` values for non-bye matchups.

2. **Handling prediction cascade in tabulation for "simple" prediction mode**
   - What we know: In "advanced" prediction mode, students click the bracket diagram and `usePredictionCascade` propagates picks. In "simple" mode, students use a form-based list with the same cascade. Both modes require all matchups to be predicted before submit.
   - What's unclear: Whether students can submit partial predictions (some matchups unpredicted) and how the tabulation handles them.
   - Recommendation: The existing `allSelected` gate prevents submission without all matchups predicted. However, the tabulation should be defensive: handle the case where a matchup has zero predictions by flagging it as 'no_predictions' for teacher resolution. This is already planned in the locked decisions.

3. **Broadcast event type for reveal**
   - What we know: Existing broadcast types are defined in `broadcast.ts` as a union type. New types need to be added.
   - What's unclear: Whether to add new event types to the existing `BracketUpdateType` union or create separate events.
   - Recommendation: Add to existing union: `'reveal_round' | 'reveal_complete' | 'results_prepared'`. This keeps all bracket events on the same channel (`bracket:{bracketId}`) and lets existing hooks handle them with minimal changes.
