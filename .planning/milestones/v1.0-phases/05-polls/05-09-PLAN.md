The bug: In `client.tsx`, `handleStatusChange('closed')` calls `updatePollStatus` (which broadcasts `poll_closed`), then calls `router.refresh()`. The refresh re-renders the server component with `poll.status='closed'`. In `poll-results.tsx`, `prevStatusRef` (line 46) initializes to `poll.status`, so after refresh it starts as `'closed'`. When the broadcast arrives setting `liveStatus` to `'closed'`, the `active->closed` transition in the useEffect (lines 54-59) is never detected, so `showReveal` never becomes true.

Fix approach -- track closing intent locally instead of relying on status transition detection:

**In `client.tsx`:**
1. Add a `forceReveal` state: `const [forceReveal, setForceReveal] = useState(false)`
2. In `handleStatusChange`, when `newStatus === 'closed'`, set `setForceReveal(true)` after line 71 (`setCurrentStatus(newStatus)`), before line 72 (`router.refresh()`)
3. Pass `forceReveal={forceReveal}` and `onRevealDismissed={() => setForceReveal(false)}` as props to `PollResults` (the `resultsContent` JSX at lines 77-84)

**In `poll-results.tsx`:**
1. Add to the `PollResultsProps` interface:
   - `forceReveal?: boolean`
   - `onRevealDismissed?: () => void`
2. Accept these new props in the component destructuring
3. REMOVE the `prevStatusRef` (line 46) and the status transition detection `useEffect` (lines 54-59) entirely -- this approach is inherently racy
4. Instead, add a useEffect that triggers reveal from the `forceReveal` prop:
   ```typescript
   useEffect(() => {
     if (forceReveal) {
       setShowReveal(true)
     }
   }, [forceReveal])
   ```
5. Update the `onDismiss` callback on `PollReveal` (line 198) to also call `onRevealDismissed?.()`:
   ```typescript
   onDismiss={() => {
     setShowReveal(false)

   ```

This eliminates the race entirely: the parent component knows the teacher clicked "Close Poll" and directly tells the results component to show the reveal, regardless of broadcast timing.

The bug: `getSimplePollVoteCounts` (poll.ts lines 332-345) uses `prisma.pollVote.groupBy` which only returns groups with votes > 0. When a student changes their vote from Option A to Option B, Option A may drop to 0 votes and be omitted from the result. The broadcast sends this sparse object `{optionB: 2}`. In `use-realtime-poll.ts` line 80, the batch flush does `{...prev, ...pending}` (spread merge), which preserves the stale `{optionA: 1}` from previous state. Result: Option A still shows 1 vote when it should show 0.

Fix in TWO places (belt and suspenders):

**Fix 1 - DAL returns complete counts (src/lib/dal/poll.ts):**
Modify `getSimplePollVoteCounts` to return zero counts for ALL poll options, not just options with votes.

Replace the function (lines 332-345) with:
```typescript
export async function getSimplePollVoteCounts(
  pollId: string
): Promise<Record<string, number>> {
  // Fetch all option IDs for this poll to ensure zeros are included
  const options = await prisma.pollOption.findMany({
    where: { pollId },
    select: { id: true },
  })

  const groups = await prisma.pollVote.groupBy({
    by: ['optionId'],
    where: { pollId, rank: 1 },
    _count: { id: true },
  })

  // Start with all options at zero, then overlay actual counts
  const counts: Record<string, number> = {}
  for (const opt of options) {
    counts[opt.id] = 0
  }
  for (const g of groups) {
    counts[g.optionId] = g._count.id
  }
  return counts
}
```

This ensures the broadcast payload always includes `{optionA: 0, optionB: 2}` instead of `{optionB: 2}`.

**Fix 2 - Full replacement flush in real-time hook (src/hooks/use-realtime-poll.ts):**
Change ONLY the batch flush merge from spread (additive) to full replacement.

In the `flushInterval` callback (lines 73-87), change line 80 from:
```typescript
setVoteCounts((prev) => ({ ...prev, ...pending }))
```
to:
```typescript
setVoteCounts(pending)
```

This ensures that even if broadcast data is complete (with zeros), we do a clean replacement rather than merging over stale state. The broadcast always contains ALL option counts now, so full replacement is correct.

**DO NOT change line 99** (`pendingVoteCounts.current = { ...pendingVoteCounts.current, ...counts }`). The spread merge accumulation at line 99 is correct and necessary for batching: if multiple broadcasts arrive within the batch window, each broadcast contains complete state, but the spread merge ensures the latest complete snapshot overwrites earlier keys while preserving the accumulation pattern. Changing this to simple assignment would cause data loss if a broadcast arrives between flush intervals and another broadcast overwrites it before the next flush.

**No changes needed in `src/actions/poll.ts` or `src/app/api/polls/[pollId]/state/route.ts`** -- they both call `getSimplePollVoteCounts` which now returns complete counts. The fix propagates automatically through the existing call chain (poll.ts line 364 in castPollVote action, and state/route.ts line 41).
