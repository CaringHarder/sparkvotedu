  <action>
Apply three targeted fixes to eliminate stale bracket state after round advancement:

**Fix 1 -- API route force-dynamic (src/app/api/brackets/[bracketId]/state/route.ts):**
Add `export const dynamic = 'force-dynamic'` at the top of the file (after imports, before the GET function). This prevents Next.js 16 from caching GET responses on this route. The route already reads from Prisma on every request but Next.js may cache the response at the framework level without this directive.

**Fix 2 -- Cache-busted fetch with timestamp (src/hooks/use-realtime-bracket.ts):**
In the `fetchBracketState` callback, change the fetch call from:
```
fetch(`/api/brackets/${bracketId}/state`)
```
to:
```
fetch(`/api/brackets/${bracketId}/state?t=${Date.now()}`, { cache: 'no-store' })
```
This prevents both browser-level fetch deduplication (timestamp makes each URL unique) and Next.js client-side fetch caching (`cache: 'no-store'`). Belt-and-suspenders with Fix 1.

**Fix 3 -- Stale response sequence guard (src/hooks/use-realtime-bracket.ts):**
Add a `fetchSeqRef = useRef(0)` alongside the existing refs. At the start of `fetchBracketState`, increment `fetchSeqRef.current` and capture the value as `const seq = ++fetchSeqRef.current`. After `await res.json()`, check `if (seq !== fetchSeqRef.current) return` to discard out-of-order responses. This prevents the race condition where multiple rapid `bracket_update` events (e.g., `winner_selected` + `voting_opened` in quick succession) trigger overlapping fetches and an older response overwrites a newer one.

**What NOT to change:**
- Do NOT modify the batch flush interval logic (it works correctly)
- Do NOT modify the Supabase channel subscription (it is bracket-scoped and correct)
- Do NOT touch celebration, completion detection, or bracket advancement logic
- Do NOT modify live-dashboard.tsx, bracket-diagram.tsx, or any other file -- the voteLabels computation logic is correct; it just needs correct input data from fetchBracketState

**Why these three fixes together:** The research identified that the final round fails because after round advancement, `fetchBracketState` returns stale/cached data where the final matchup still shows status "pending" with null entrants. The voteLabels useMemo only generates labels for matchups with status "voting" or "decided", so stale "pending" status means no vote labels appear. Fix 1+2 ensure fresh data; Fix 3 ensures rapid-fire events don't corrupt state with out-of-order responses.
