<objective>
Fix stale participation denominator when students join a session while a poll is active.

Purpose: UAT Test 3 revealed that the teacher dashboard "X of Y voted" denominator does not update when a new student joins -- it only updates after the student submits a vote. The root cause is that `joinSessionByName` creates a participant but emits no broadcast, and `useRealtimePoll` has no listener for participant-join events.

Output: Three files modified so that new participant joins trigger a broadcast that the teacher's poll live dashboard receives, causing a re-fetch of participantCount.
</objective>

1. Add an optional `sessionId` parameter to `useRealtimePoll`: change signature from `useRealtimePoll(pollId: string, batchIntervalMs = 2000)` to `useRealtimePoll(pollId: string, sessionId?: string | null, batchIntervalMs = 2000)`.

2. Inside the `useEffect`, AFTER the existing `poll:{pollId}` channel subscription block, add a SECOND channel subscription for `activities:{sessionId}` -- but ONLY if `sessionId` is truthy. Subscribe to the `participant_joined` event on this channel. When received, call `fetchPollState()` to re-fetch the participantCount from the API (which already queries the DB for the current non-banned count).

3. The activities channel subscription should follow this pattern:
   ```
   let activitiesChannel: ReturnType<typeof supabase.channel> | null = null
   if (sessionId) {
     activitiesChannel = supabase
       .channel(`poll-participants:${sessionId}`)
       .on('broadcast', { event: 'participant_joined' }, () => {
         fetchPollState()
       })
       .subscribe()
   }
   ```
   Use channel name `poll-participants:{sessionId}` (NOT `activities:{sessionId}`) to avoid conflicting with the existing `useRealtimeActivities` hook that also subscribes to `activities:{sessionId}`. Supabase client multiplexes channels by name -- using the same name would share the subscription but that is fine since they listen on different events. However, using a distinct client-side channel name is cleaner. The TOPIC for broadcast matching is set server-side; the client channel name is just a local identifier.

   IMPORTANT CORRECTION: Actually, Supabase Broadcast channels must match exactly between sender and receiver. The server broadcasts to topic `activities:{sessionId}`, so the client MUST subscribe to channel `activities:{sessionId}` to receive it. Use `activities:{sessionId}` as the channel name. The existing `useRealtimeActivities` hook listens for `activity_update` events on this channel, while this hook listens for `participant_joined` -- different events on the same channel work fine.

   Final pattern:
   ```
   let activitiesChannel: ReturnType<typeof supabase.channel> | null = null
   if (sessionId) {
     activitiesChannel = supabase
       .channel(`activities:${sessionId}`)
       .on('broadcast', { event: 'participant_joined' }, () => {
         fetchPollState()
       })
       .subscribe()
   }
   ```

4. In the cleanup return function, add `if (activitiesChannel) supabase.removeChannel(activitiesChannel)`.

5. Add `sessionId` to the useEffect dependency array: `[pollId, sessionId, supabase, batchIntervalMs, fetchPollState]`.

6. Update the call site in `src/components/poll/poll-results.tsx`: change `useRealtimePoll(poll.id)` to `useRealtimePoll(poll.id, poll.sessionId)`. The `poll` prop already contains `sessionId: string | null` from `PollWithOptions`.
