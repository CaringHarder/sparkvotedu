**Part B: Initialize revealComplete from bracket state in usePredictions**

In `use-predictions.ts`, the `revealComplete` state is initialized as `false` and only ever set to `true` from the `reveal_complete` broadcast event (line ~90-93). This means:
- If a student loads the page AFTER the reveal is already complete, revealComplete stays false
- The podium celebration in PredictionReveal only fires when revealComplete is true

Fix: Accept an optional `initialPredictionStatus` parameter in `usePredictions`:

```typescript
export function usePredictions(
  bracketId: string,
  participantId?: string,
  initialPredictionStatus?: string
) {
```

Then initialize `revealComplete` based on the initial status:
```typescript
const [revealComplete, setRevealComplete] = useState(initialPredictionStatus === 'completed')
```

**Part C: Pass initialPredictionStatus from PredictionReveal to usePredictions**

In `prediction-reveal.tsx`, the `usePredictions` call (line ~66):
```typescript
const { leaderboard, myPredictions: realtimePredictions, revealedUpToRound, revealComplete } =
  usePredictions(bracket.id, participantId)
```

Change to:
```typescript
const { leaderboard, myPredictions: realtimePredictions, revealedUpToRound, revealComplete } =
  usePredictions(bracket.id, participantId, bracket.predictionStatus ?? undefined)
```

This ensures that when a student loads a completed bracket page (cold-start), revealComplete is immediately true, which triggers the podium celebration effect (lines 92-98).

**Part D: Add fallback podium trigger for completed status on mount**

In `prediction-reveal.tsx`, the podium trigger useEffect (lines 92-98) only fires on `revealComplete` change. For extra safety, also check on mount if the bracket is already completed:

The existing effect already handles this correctly IF revealComplete is initialized properly (Part B/C). But add a second check inside the PredictionReveal component that also triggers podium if `bracket.predictionStatus === 'completed'` and the reveal data shows all rounds:

```typescript
// Fallback: show podium on cold-start completed bracket
useEffect(() => {
  if (bracket.predictionStatus === 'completed' && !podiumDismissed && !showPodium) {
    const timer = setTimeout(() => setShowPodium(true), 500)
    return () => clearTimeout(timer)
  }
}, [bracket.predictionStatus, podiumDismissed, showPodium])
```

This is a belt-and-suspenders approach: Part B/C handles the normal case via revealComplete initialization, and this handles edge cases where the hook might not initialize correctly.

Add a new optional prop to PredictiveBracket for real-time status override:
```tsx
<PredictiveBracket
  bracket={bracket}
  participantId={participantId}
  isTeacher={false}
  effectivePredictionStatus={effectiveStatus}
/>
```

**Part B: Accept and use effectivePredictionStatus in PredictiveBracket**

In `predictive-bracket.tsx`, update the props interface:
```typescript
interface PredictiveBracketProps {
  bracket: BracketWithDetails
  participantId: string
  isTeacher: boolean
  effectivePredictionStatus?: string
}
```

In the component body (line ~35), prefer the override:
```typescript
const predictionStatus = props.effectivePredictionStatus ?? bracket.predictionStatus ?? 'draft'
```

This prop flows into both the simple (SimplePredictionMode) and advanced (AdvancedPredictionMode) sub-components via the `predictionStatus` variable, which already controls `isPredictionsOpen`:

When `isPredictionsOpen` becomes false (because effectiveStatus changed from predictions_open to something else), both the SimplePredictionMode and AdvancedPredictionMode components will:
- Show the "Predictions are closed" lock banner (line ~699-714 and ~870-894)

In SimplePredictionMode, find the "Edit Predictions" button (around line ~729):

Wrap it with `{isPredictionsOpen && (` ... `)}` to ensure it never shows when predictions are closed.

Do the same for the AdvancedPredictionMode "Edit Predictions" button (around line ~909).

**Part C: Add error handling in handleSubmit**

In both SimplePredictionMode and AdvancedPredictionMode, the `handleSubmit` function (lines ~674 and ~848) has:
```typescript
if (result && 'success' in result) {
  setIsEditing(false)
  onRefetch()
}
```

Add an else branch to handle server-side rejection:
```typescript
if (result && 'success' in result) {
  setIsEditing(false)
  onRefetch()

  setIsEditing(false)
}
```

This ensures that if a student somehow has the edit form open when predictions close (race condition), the server rejection is shown to the user and editing mode is exited.
