  <name>Task 2: Create name-based server actions (joinSessionByName, claimIdentity, updateParticipantName)</name>
  <files>

  <action>
    **Update `src/actions/student.ts`:**

    **1. Add `joinSessionByName` server action:**
    - Validation schema: `{ code: z.string().regex(/^\d{6}$/), firstName: firstNameSchema }` -- import `firstNameSchema` from `@/lib/validations/first-name`
    - Step 1: Find session with `findSessionByCode` (not `findActiveSessionByCode`) to support ended sessions per locked decision
    - Step 2: If session status is 'ended', return session info with a flag `sessionEnded: true` so the UI can show results. Add `sessionEnded?: boolean` to `JoinResult` in types if not already there (or handle in this action by returning a structured response).
    - Step 3: If session is active, do case-insensitive name lookup via `findParticipantsByFirstName(session.id, firstName)`
    - Step 4: If no matches, create new participant with `createParticipant(session.id, null, undefined, firstName)` -- deviceId is null for name-based join
    - Step 5: If matches found, return `{ duplicates: existing.map(p => ({ id: p.id, funName: p.funName })), session: sessionInfo }` for disambiguation UI
    - Build `sessionInfo` the same way as existing `joinSession` does (id, code, name, status, teacherName)
    - Update `toParticipantData` helper to include `firstName` field

    **2. Add `claimIdentity` server action:**
    - Input: `{ participantId: string, sessionCode: string }`
    - Validate: find participant by ID, check not banned
    - Verify the participant belongs to the session with the given code (security: prevent claiming participants from other sessions)
    - Update `lastSeenAt` via `updateLastSeen`
    - Return `{ participant: toParticipantData(participant), session: sessionInfo, returning: true }`

    **3. Add `updateParticipantName` server action:**
    - Input: `{ participantId: string, firstName: string }`
    - Validate firstName with `firstNameSchema`
    - Look up participant to get sessionId
    - Check for name collision: `findParticipantsByFirstName(sessionId, newFirstName)` -- if any exist that are NOT the current participant, return `{ error: 'That name is already taken in this session. Try a different name.' }`
    - If no collision, update via `updateFirstName(participantId, firstName)`
    - Return updated participant data

    **4. Update `toParticipantData` helper:**
    - Add `firstName` field to the helper function's input type and output mapping
    - The Prisma return type already includes `firstName` from the schema -- just need to include it in the mapping

    **Keep existing `joinSession`, `rerollName`, `getRecoveryCode`, `recoverIdentity` actions** -- they still work and provide backward compatibility. The old `joinSession` will be deprecated but not removed.
  </action>
