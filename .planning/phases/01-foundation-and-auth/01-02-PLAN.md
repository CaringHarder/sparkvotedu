  <action>
    1. Create `src/app/proxy.ts` (NOT middleware.ts -- Next.js 16 renamed it):
       - Export `async function proxy(request: NextRequest)` (NOT `middleware`).
       - Create a Supabase server client using `createServerClient` from `@supabase/ssr`.
       - Use the `getAll()`/`setAll()` cookie pattern (per research). NEVER use individual get/set/remove.
       - Call `supabase.auth.getClaims()` to validate the JWT (NOT getSession() -- security risk per research).
       - Define auth pages: /login, /signup, /forgot-password, /update-password, /auth.
       - Define public pages: / (root landing page).
       - If no valid claims AND request is not to an auth page or public page, redirect to /login.
       - If valid claims AND request is to an auth page, redirect to /dashboard.
       - Otherwise, pass through with the supabaseResponse (which carries refreshed cookies).
       - Export `config` with matcher that excludes static files: `/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)`.

    2. Create `src/app/auth/callback/route.ts`:
       - Export async GET handler.
       - Extract `code` and `next` (default `/dashboard`) from searchParams.
       - If code exists, call `supabase.auth.exchangeCodeForSession(code)`.
       - On success, redirect to `next` path.
       - On failure or missing code, redirect to `/login?error=auth_callback_failed`.
       - This route handles BOTH OAuth callbacks and password reset callbacks.

    3. Create `src/actions/auth.ts` with Server Actions:
       - `signUp(formData: FormData)`: Validate with signUpSchema from validation.ts. Call `supabase.auth.signUp()` with email, password, and `options.data.name`. Return `{ success }` or `{ error }`.
       - `signIn(formData: FormData)`: Call `supabase.auth.signInWithPassword()` with email and password. On success, `redirect('/dashboard')`. On error, return `{ error }`.
       - Mark both with `'use server'` at top of file.
       - Import createClient from `@/lib/supabase/server`.
       - Import validation schemas from `@/lib/utils/validation`.

    4. Create `src/lib/dal/auth.ts` -- Data Access Layer:
       - Export `async function getAuthenticatedTeacher()`:
         - Create Supabase server client.
         - Call `getClaims()` to get the authenticated user's claims.
         - If no claims/error, return null.
         - Query Prisma for Teacher by `supabaseAuthId` matching `claims.sub`.
         - If no Teacher found, CREATE one with email from claims, name from `claims.user_metadata?.name`, supabaseAuthId from `claims.sub`. This is the "ensure teacher exists" pattern from research (simpler than database triggers).
         - Return the Teacher record.
       - This function is the AUTHORITATIVE auth check. Proxy is optimistic only.

    IMPORTANT: Use `getClaims()` everywhere (NOT `getSession()`). The proxy is a UX convenience, NOT a security boundary. The DAL is the authoritative check.
    IMPORTANT: The `'use server'` directive goes at the top of actions/auth.ts, not on individual functions.
  </action>
  <verify>
    - `npx tsc --noEmit` passes with no type errors
    - proxy.ts exports a function named `proxy` (not `middleware`)
    - proxy.ts uses `getClaims()` (grep for getClaims)
    - auth callback route exports GET handler
    - actions/auth.ts has 'use server' directive
    - dal/auth.ts imports from prisma.ts and supabase/server.ts
  </verify>
  <done>Proxy intercepts all non-static requests, validates JWT with getClaims(), redirects appropriately. Auth callback handles PKCE code exchange. Server actions handle signup and signin with Zod validation. DAL provides authoritative teacher record lookup with auto-creation on first login.</done>
