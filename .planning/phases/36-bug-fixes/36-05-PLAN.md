**Root cause (from research):** For SE/DE brackets, `currentVoterIds` computation requires `selectedMatchupId` to be set. It defaults to `null`, so the sidebar always returns an empty voter list -- green dots never appear.

**Fix the `currentVoterIds` useMemo** (around line 787-811) for SE/DE bracket types:

When `selectedMatchupId` is null (no matchup explicitly selected by teacher):
1. Find all currently voting matchups in the current round:
   ```typescript
   const votingMatchups = currentMatchups.filter(
     m => m.status === 'voting' && m.round === currentRound
   )
   ```
2. Union voter IDs across all voting matchups:
   ```typescript
   const allVoterIds = new Set<string>()
   for (const m of votingMatchups) {
     for (const id of (mergedVoterIds[m.id] ?? [])) {
       allVoterIds.add(id)
     }
   }
   return [...allVoterIds]
   ```
3. When `selectedMatchupId` IS set (teacher clicked a specific matchup), keep existing behavior: return `mergedVoterIds[selectedMatchupId] ?? []`

**This applies to both SE and DE bracket types** (they share the same code path per research line 807-809).

**RR and Predictive brackets** already work correctly (RR uses round-based intersection, Predictive uses 'predictions' key) -- do NOT modify their logic.

**Variable naming:** Use `votingMatchups` and `allVoterIds` for clarity. The union approach means a student gets a green dot if they voted on ANY matchup in the current round (per research recommendation).

**Determine `currentRound`:** Check how `currentRound` is computed in the live-dashboard component. It should already be available as state or derived from bracket data. If not directly available, derive it from the voting matchups: `const currentRound = Math.max(...currentMatchups.filter(m => m.status === 'voting').map(m => m.round), 0)`.

**Determine `currentMatchups`:** This should already be available in the component. Check variable names -- it may be `matchups`, `bracket.matchups`, or similar. Use what exists.
