**1. `src/lib/dal/class-session.ts`** -- Class session data access

Import prisma from `@/lib/prisma` and generateClassCode from `@/lib/student/class-codes`.

Export these functions:

- `createClassSession(teacherId: string, name?: string): Promise<ClassSession>`
  - Call generateClassCode() to get a unique 6-digit code
  - Create ClassSession with code, status: 'active', teacherId, optional name
  - Return created session

- `findActiveSessionByCode(code: string): Promise<ClassSession & { teacher: { name: string | null } } | null>`
  - Find first ClassSession where code matches AND status is 'active'
  - Include teacher name for welcome screen display
  - Return null if not found

- `getSessionWithParticipants(sessionId: string, teacherId: string): Promise<ClassSession & { participants: StudentParticipant[] } | null>`
  - Find session by ID, verify teacherId matches (authorization check)
  - Include all participants ordered by createdAt
  - Return null if not found or unauthorized

- `endSession(sessionId: string, teacherId: string): Promise<ClassSession>`
  - Update session status to 'ended', set endedAt to now()
  - Verify teacherId matches before updating (authorization)
  - Throw error if session not found or unauthorized

- `getTeacherSessions(teacherId: string): Promise<ClassSession[]>`
  - Find all sessions for teacher, ordered by createdAt desc
  - Include participant count (use _count)

**2. `src/lib/dal/student-session.ts`** -- Student participant data access

Import prisma from `@/lib/prisma`, generateFunName from `@/lib/student/fun-names`, nanoid from `nanoid`.

Export these functions:

- `findParticipantByDevice(sessionId: string, deviceId: string): Promise<StudentParticipant | null>`
  - Use the @@unique([sessionId, deviceId]) compound key for lookup
  - Return null if not found

- `findParticipantByFingerprint(sessionId: string, fingerprint: string): Promise<StudentParticipant | null>`
  - Find first participant matching sessionId + fingerprint where banned is false
  - Return null if not found

- `findParticipantByRecoveryCode(recoveryCode: string): Promise<StudentParticipant & { session: ClassSession } | null>`
  - Find participant by recoveryCode, include session
  - Return null if not found

- `createParticipant(sessionId: string, deviceId: string, fingerprint?: string): Promise<StudentParticipant>`
  - Fetch all existing fun names for the session
  - Call generateFunName(existingNameSet) to get unique name
  - Create StudentParticipant with sessionId, funName, deviceId, fingerprint
  - Return created participant

- `updateParticipantDevice(participantId: string, deviceId: string): Promise<StudentParticipant>`
  - Update the deviceId field (used when localStorage was cleared but fingerprint matched)
  - Also update lastSeenAt to now()

- `rerollParticipantName(participantId: string): Promise<StudentParticipant>`
  - Find participant, verify rerollUsed is false, throw if already used
  - Fetch existing names in session, generate new fun name
  - Update participant with new funName and rerollUsed: true
  - Return updated participant

- `generateRecoveryCode(participantId: string): Promise<string>`
  - Find participant, if recoveryCode already exists return it
  - Generate new code with `nanoid(8).toUpperCase()` -- 8 chars, uppercase alphanumeric
  - Update participant with recoveryCode
  - Return the recovery code

- `banParticipant(participantId: string): Promise<StudentParticipant>`
  - Update participant with banned: true

- `removeParticipant(participantId: string): Promise<void>`
  - Delete participant from database

All DAL functions use the prisma singleton. No direct Supabase auth checks in student DAL -- students are unauthenticated. Teacher DAL functions verify ownership via teacherId parameter (caller passes from auth).

**1. `src/actions/student.ts`** -- Student-facing server actions (NO auth required)

Add `'use server'` directive. Import Zod for validation, DAL functions from student-session.ts and class-session.ts.

- `joinSession(input: { code: string; deviceId: string; fingerprint?: string }): Promise<JoinResult>`
  - Validate with Zod: code must be exactly 6 digits (`/^\d{6}$/`), deviceId must be non-empty string
  - Step 1: Find active session by code. Return error "Invalid or expired class code" if not found.
  - Step 2: Check deviceId match (findParticipantByDevice). If found AND not banned, return existing participant with `returning: true`. If found AND banned, return error "You have been removed from this session".
  - Step 3: Check fingerprint match (findParticipantByFingerprint) if fingerprint provided. If found, update deviceId (localStorage was cleared), return with `returning: true`.
  - Step 4: New student -- create participant via createParticipant. Return with `returning: false`.
  - Return shape: `{ participant, session: { id, code, teacherName }, returning }` or `{ error }`
  - Follow the EXACT pattern from 02-RESEARCH.md "Server Action: Join a Session"

- `rerollName(participantId: string): Promise<{ participant?: StudentParticipantData; error?: string }>`
  - Call rerollParticipantName from DAL
  - Catch error if reroll already used, return `{ error: 'You already used your reroll' }`
  - Return updated participant data

- `getRecoveryCode(participantId: string): Promise<{ recoveryCode?: string; error?: string }>`
  - Call generateRecoveryCode from DAL
  - Return the recovery code string

- `recoverIdentity(recoveryCode: string, deviceId: string): Promise<JoinResult>`
  - Validate: recoveryCode non-empty, deviceId non-empty
  - Find participant by recovery code via DAL
  - If not found, return error "Invalid recovery code"
  - If found, update deviceId to new one, return participant with returning: true
  - Optionally: invalidate old recovery code and generate new one (single-use per research recommendation)

**2. `src/actions/class-session.ts`** -- Teacher-facing session actions (auth REQUIRED)

Add `'use server'` directive. Import DAL functions from class-session.ts and student-session.ts. Import `getAuthenticatedTeacher` from `@/lib/dal/auth.ts` (the existing auth DAL).

- `createSession(name?: string): Promise<{ session?: ClassSessionData; error?: string }>`
  - Call getAuthenticatedTeacher() to get current teacher (throws if not authenticated)
  - Call createClassSession(teacher.id, name) from DAL
  - Return session data

- `endSession(sessionId: string): Promise<{ success?: boolean; error?: string }>`
  - Call getAuthenticatedTeacher()
  - Call endSession(sessionId, teacher.id) from DAL
  - Return success

- `removeStudent(participantId: string): Promise<{ success?: boolean; error?: string }>`
  - Call getAuthenticatedTeacher() for authorization
  - Call removeParticipant from DAL
  - Return success

- `banStudent(participantId: string): Promise<{ success?: boolean; error?: string }>`
  - Call getAuthenticatedTeacher() for authorization
  - Call banParticipant from DAL
  - Return success

NOTE: Check how `getAuthenticatedTeacher` is named/exported in the existing `src/lib/dal/auth.ts`. It might be named differently (e.g., `getCurrentTeacher`, `getTeacher`). Read the existing auth DAL and use whatever function provides the authenticated teacher record. If it doesn't exist, the pattern is: call getClaims() from supabase, find teacher by supabaseAuthId. Create a utility if needed.

**3. Update `src/app/proxy.ts`** -- Add student routes to public pages

The existing proxy logic should remain unchanged -- authenticated users still get redirected away from auth pages, and unauthenticated users still get redirected to login for protected pages. The only change is that student routes are now treated as public (no auth required).

Follow the pattern from 02-RESEARCH.md Pattern 5.
