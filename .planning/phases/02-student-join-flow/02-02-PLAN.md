Install the device fingerprinting library and build the composite device identity system. This plan creates the client-side infrastructure that identifies students across sessions without requiring accounts -- using localStorage UUID as the primary signal and FingerprintJS as the secondary signal.

Purpose: Device identity is what makes anonymous student recognition work. Without it, returning students get new fun names every time. This is the "invisible login" that makes the product feel magical.
Output: FingerprintJS installed, fingerprint utility, localStorage identity utility, React hook exposing composite device identity.

**Step 2: Create `src/lib/student/session-identity.ts`**

This is the PRIMARY identity layer -- a UUID stored in localStorage. On school Chromebooks, each student's Chrome profile has its own localStorage, making this the most reliable identifier even on identical hardware.

- Define `IDENTITY_KEY = 'sparkvotedu_device_id'` constant (exported for testing)
- Export `getOrCreateDeviceId(): string`
  - Guard: if `typeof window === 'undefined'` return empty string (SSR safety)
  - Check `localStorage.getItem(IDENTITY_KEY)` for existing ID
  - If not found, generate new UUID via `crypto.randomUUID()`, store it, return it
  - If found, return existing ID
- Export `clearDeviceId(): void` -- removes the identity key from localStorage (for testing/debug)

Follow the exact pattern from 02-RESEARCH.md Pattern 1 (session-identity.ts section).

**Step 3: Create `src/lib/student/fingerprint.ts`**

This is the SECONDARY identity layer -- FingerprintJS collects canvas, WebGL, audio context, font, and other signals to produce a browser fingerprint hash. On identical school hardware this may collide (1-3% rate), which is why it's secondary to localStorage.

- Import FingerprintJS from `@fingerprintjs/fingerprintjs`
- Cache the FingerprintJS load promise in a module-level variable to avoid re-initialization
- Export `getBrowserFingerprint(): Promise<string>`
  - Guard: if `typeof window === 'undefined'` return empty string (SSR safety)
  - Lazy-load FingerprintJS (call `FingerprintJS.load()` once, cache the promise)
  - Call `fp.get()` to compute fingerprint
  - Return `result.visitorId` string
- Wrap in try/catch -- if FingerprintJS fails (blocked by browser/extension), return empty string (graceful degradation)

Follow the exact pattern from 02-RESEARCH.md Pattern 1 (fingerprint.ts section).

Create `src/lib/student/` directory if it doesn't already exist (it may have been created by Plan 02-01 if running in same wave).

  <action>
Create `src/hooks/use-device-identity.ts` -- a client-side React hook that combines both identity layers into a single composite payload.

- Add `'use client'` directive at top of file
- Import `useEffect`, `useState` from 'react'
- Import `getOrCreateDeviceId` from `@/lib/student/session-identity`
- Import `getBrowserFingerprint` from `@/lib/student/fingerprint`
- Import `DeviceIdentity` type from `@/types/student` (created in Plan 02-01)

Export `useDeviceIdentity(): DeviceIdentity` hook:
- Initialize state with `{ deviceId: '', fingerprint: '', ready: false }`
- In useEffect (runs client-side only):
  - Call `getOrCreateDeviceId()` synchronously (localStorage is sync)
  - Call `await getBrowserFingerprint()` asynchronously
  - Set state to `{ deviceId, fingerprint, ready: true }`
- Return the identity state object
- The `ready` flag lets consumers know when identity is fully computed (fingerprint is async)

IMPORTANT: This hook MUST be used only in Client Components. Server Components cannot access localStorage or run FingerprintJS.

If `src/types/student.ts` doesn't exist yet (Plan 02-01 may not have run), define DeviceIdentity interface locally in this file as a fallback, with a TODO comment to import from types when available. However, if the type file exists, import from it.
