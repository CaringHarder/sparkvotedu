# Phase 31.1: Activity Card Layout Fix & Quick Settings Toggle - Research

**Researched:** 2026-03-01
**Domain:** CSS Grid layout, Radix UI Switch, server actions, real-time broadcast
**Confidence:** HIGH

## Summary

This phase addresses two distinct but related teacher UX issues: (1) activity cards (brackets and polls) truncating titles and descriptions when many activities exist at any screen size, and (2) adding inline quick settings toggles on bracket detail/edit and live dashboard pages (viewing mode for brackets) and poll detail/live dashboard pages (show live results, allow vote change).

The card layout fix is a CSS-only change. All three card grid containers (`BracketCardList`, `PollCardList`, `ActivitiesList`) use `grid grid-cols-1 gap-4 md:grid-cols-2 lg:grid-cols-3` which forces 3 columns on large screens regardless of how many cards exist or how wide the content needs. The fix: use CSS Grid `auto-fill` with a minimum card width so the grid naturally drops columns rather than squeezing cards.

The quick settings toggle work is a thin feature layer. The `@radix-ui/react-switch` package is already installed and a `Switch` component already exists at `src/components/ui/switch.tsx`. Server actions and DAL functions for updating both bracket viewingMode (via `prisma.bracket.update`) and poll settings (`updatePoll` / `updatePollDAL` for `allowVoteChange` / `showLiveResults`) are partially in place. The main work is: (a) creating a new `updateBracketViewingMode` server action (no bracket-level generic update action exists yet), (b) wiring toggle UI on 4 pages, and (c) broadcasting setting changes to students via existing Supabase Broadcast infrastructure.

**Primary recommendation:** Fix card grids with `grid-template-columns: repeat(auto-fill, minmax(320px, 1fr))` and use the existing Switch component plus existing server actions for quick toggles, adding only the missing `updateBracketViewingMode` action.

### Pattern 1: Auto-Fill Grid with Minimum Width
**What:** Replace hardcoded responsive breakpoint columns with CSS Grid auto-fill
**When to use:** Card grids that need to work at any viewport width with variable content counts
**Example:**
```css
/* Before (current -- forces 3 cols at lg regardless of content) */
.grid.grid-cols-1.md\:grid-cols-2.lg\:grid-cols-3 { }

/* After -- cards have min 320px width, grid auto-adjusts column count */
.grid { grid-template-columns: repeat(auto-fill, minmax(320px, 1fr)); }
```
In Tailwind v4, this is expressed as an arbitrary value:
```html
<div class="grid gap-4 grid-cols-[repeat(auto-fill,minmax(320px,1fr))]">
```

### Pattern 2: Labeled Inline Switch
**What:** A Switch with a visible label, matching the existing pause/resume pattern on the live dashboard
**When to use:** Binary toggles that need to blend into the page layout
**Example:**

### Anti-Patterns to Avoid
- **Hardcoded column counts at breakpoints:** Using `md:grid-cols-2 lg:grid-cols-3` forces columns regardless of card content width. Use auto-fill minmax instead.
- **Creating a separate "Quick Settings" section/drawer:** Decisions say toggles should blend into the existing layout, not be in their own section.
- **Putting toggles on activity cards:** Explicitly out of scope per decisions.

## Common Pitfalls

### Pitfall 1: Tailwind v4 Arbitrary Grid Values
**What goes wrong:** Tailwind v4 arbitrary value syntax differs slightly from v3. The `grid-cols-[...]` utility needs to be tested.
**Why it happens:** Tailwind v4 uses a new engine and some arbitrary value syntaxes have changed.
**How to avoid:** Test the exact class `grid-cols-[repeat(auto-fill,minmax(320px,1fr))]` in the dev server. If it doesn't work, use an inline `style` attribute as fallback.
**Warning signs:** Cards render in a single column or don't respond to width changes.

### Pitfall 2: Poll allowVoteChange Toggle During Active Session
**What goes wrong:** Toggling `allowVoteChange` on a live poll must take effect immediately for students who are currently on the poll page.
**Why it happens:** Students fetch poll settings on page load via `/api/polls/[pollId]/state`. If the setting changes mid-session, students won't see it until they refetch.
**How to avoid:** After updating the database, broadcast a `poll_settings_changed` event on the `poll:{pollId}` channel. The student poll page already listens to `poll_update` events via `useRealtimePoll` hook, but need to verify it refetches the full state (including allowVoteChange) on those events.
**Warning signs:** Student can still change vote after teacher disables it (until page refresh).

### Pitfall 3: Bracket viewingMode Change Not Reflected for Students
**What goes wrong:** Teacher toggles simple<->advanced on live dashboard, but students don't see the change.
**Why it happens:** The student bracket page fetches state once on load (and on `bracket_update` events). The `useRealtimeBracket` hook does a full refetch on `bracket_update`, which includes `viewingMode` from the state API. But a new broadcast event type needs to be used.
**How to avoid:** After updating viewingMode in the database, broadcast a `bracket_update` event (e.g. type `settings_changed`). The student's `useRealtimeBracket` hook already refetches full state on any `bracket_update`, so the viewingMode will automatically update. However, the student page currently uses `bracket.viewingMode` from the initial state, not from the realtime matchup data. Need to verify the refetch path.
**Warning signs:** Student stays in simple mode after teacher switches to advanced (or vice versa).

### Pitfall 4: Card Description line-clamp Already in Place
**What goes wrong:** Assuming descriptions don't have line clamping.
**Why it happens:** The bracket card already has `line-clamp-2` on descriptions (line 226 of bracket-card.tsx). But the poll card doesn't show descriptions at all.
**How to avoid:** Verify each card component's description handling. BracketCard: already has `line-clamp-2` on `bracket.description`. PollCard: shows no description (polls use `question` as the title). ActivitiesList `ActivityItemCard`: no description shown.
**Warning signs:** Descriptions overflow on one card type but not another.

### Pitfall 5: Badge Overflow on Narrow Cards
**What goes wrong:** The badges row in bracket-card.tsx (viewing mode, bracket type, status, etc.) is a `flex` container with `gap-1.5` and no wrapping. On narrow cards, badges stack horizontally and overflow.
**Why it happens:** Line 165-221 of bracket-card.tsx shows badges in `flex items-center gap-1.5` without `flex-wrap`.
**How to avoid:** Add `flex-wrap` to the badges container. Verify badges don't visually break when wrapping to a second line.
**Warning signs:** Badges are cut off on the right edge of cards.

## Discretion Recommendations

### Responsive Breakpoint Strategy
**Recommendation:** Use `grid-cols-[repeat(auto-fill,minmax(320px,1fr))]` for all card grids. This gives:
- 320-639px: 1 column (single column on mobile)
- 640-959px: 2 columns
- 960-1279px: 3 columns
- 1280px+: 4 columns (on very wide screens with many activities)

Cards fill their column width (they grow to fit). The 320px floor ensures no card is ever unreadably narrow. This is the "fill width with a floor" approach, which is better than fixed-size cards because it uses available space.

### Toggle Control Style
**Recommendation:** Use the existing Radix Switch component (same as pause/resume toggle). Segmented controls are better for 3+ options; for binary toggles, Switch is standard. The project already uses Switch for pause/resume, so visual consistency is maintained.

### Whether simple->advanced Auto-Adjusts Related Settings
**Recommendation:** No auto-adjustment. The viewing mode toggle should ONLY change viewingMode. Seeds, vote counts, and timer are separate settings that stay on the full settings page per the locked decision. Changing mode should not have side effects.

### Whether Students See Viewing Mode Changes in Real Time
**Recommendation:** Yes, in real-time. Broadcast a `settings_changed` event and have the student bracket page refetch full state. The infrastructure is already there (useRealtimeBracket refetches on bracket_update). This requires adding the viewingMode to what the hook exposes after refetch. This is a small change.

### Confirmation UX After Toggle
**Recommendation:** No feedback needed. The toggle itself provides instant visual feedback (switch flips). This matches the pause/resume toggle pattern which also has no toast/confirmation. For poll settings that affect students immediately, the toggle label should update to reflect the new state (e.g., "Vote changes: Allowed" -> "Vote changes: Locked").

### Exact Placement on Detail/Edit Page
**Recommendation:** Place toggles in the metadata area, right after the `BracketMetadataBar` / `PollMetadataBar` component. This is "near top" and blends naturally into the existing layout without creating a new section. For polls, replace the read-only settings display (currently at lines 343-359 of poll-detail-view.tsx) with active toggles.
