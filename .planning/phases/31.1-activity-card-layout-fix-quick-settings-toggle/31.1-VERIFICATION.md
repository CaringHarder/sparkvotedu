## Human Verification Required

### 1. Card Grid Reflow Behavior

**Test:** Open the brackets list, polls list, and activities list pages. Resize the browser window from 1400px wide down to 375px.
**Expected:** Cards reflow smoothly from 4 columns to 3, then 2, then 1; no card ever narrows below approximately 320px; all card titles remain fully readable.
**Why human:** CSS Grid auto-fill column transitions are a visual/browser-rendered behavior that cannot be confirmed by grep.

### 2. Bracket Card Badge Wrapping

**Test:** On the brackets list, find a bracket card with multiple type/status/mode badges. Narrow the browser to a width where the card would previously overflow.
**Expected:** Badges wrap to a second line instead of overflowing or being clipped.
**Why human:** `flex-wrap` rendering requires visual confirmation at specific viewport widths.

### 3. Bracket Detail Page - Viewing Mode Toggle

**Test:** Navigate to a single_elimination bracket detail page. Observe the toggle. Flip it. Refresh the page.
**Expected:** Toggle appears labeled "Simple Mode" or "Advanced Mode" with an Eye icon. After flipping and refreshing, the new mode persists. Navigate to a double elimination or round robin bracket detail page - toggle should NOT appear.
**Why human:** DB persistence of the update and bracketType guard behavior require an interactive session.

### 4. Bracket Live Dashboard - Viewing Mode Toggle

**Test:** Go Live on a single_elimination bracket. Look near the pause/resume toggle in the header.
**Expected:** A viewing mode toggle appears. Flipping it changes the mode and persists.
**Why human:** Requires an active live bracket session.

### 5. Poll Detail Page - Settings Toggles

**Test:** Navigate to any poll's detail page. Look for "Show Live Results" and "Allow Vote Change" toggles.
**Expected:** Both toggles appear with labels and icons. Flipping each persists the change after page refresh.
**Why human:** DB persistence requires an interactive session.

### 6. Poll Live Dashboard - Real-Time Student Effect

**Test:** Go Live on a poll. In the teacher tab, toggle "Allow Vote Change" off. In a student tab (without refreshing), attempt to change a previously cast vote.
**Expected:** The student is immediately prevented from changing their vote (without a page refresh) because the `poll_settings_changed` broadcast triggers a hook refetch on the student side.
**Why human:** Real-time cross-tab behavior cannot be verified programmatically.
