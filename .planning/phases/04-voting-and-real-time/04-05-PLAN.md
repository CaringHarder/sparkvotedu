<objective>
Build the teacher's live dashboard for monitoring and controlling voting during an active bracket session. The dashboard shows real-time vote counts, student participation status, and provides controls for opening voting, setting timers, and advancing rounds.

Purpose: This is the teacher's command center during a live bracket session -- where they monitor student engagement, control the pace of voting, and advance the tournament. Real-time vote count updates keep the teacher informed, and the participation grid shows who has and hasn't voted.
Output: Live dashboard page at /brackets/[bracketId]/live with vote monitoring, participation tracking, and advancement controls.
</objective>

    Collapsible sidebar showing student participation status per matchup.

    Structure:

    - Grid of student tiles (4 columns on large screens, 2 on smaller):
      - Each tile: fun name, voted/not-voted icon
      - Color coding:
        - Green border/bg: voted on the selected matchup
        - Default/neutral: connected but hasn't voted yet
        - Grey/muted + opacity: disconnected (not in `connectedIds` set)
      - Tiles show the fun name only (vote choice stays private per CONTEXT.md)
    - Summary at top: "{X}/{Y} voted" fraction for the selected matchup
    - "All voted!" badge (green, animated appearance) when X === Y (100% participation)

    - Disconnected students stay in the grid (greyed out, don't disappear -- CONTEXT.md decision)

    **2. Create src/components/teacher/round-advancement-controls.tsx** (`'use client'`):

    Controls panel for managing voting and advancing the bracket.

    Props: `{ bracketId: string; matchups: MatchupData[]; selectedMatchupId: string | null; voteCounts: Record<string, VoteCounts>; currentRound: number; onSelectMatchup: (id: string) => void }`

    Structure:
    - Round selector: buttons for each round (Round 1, Quarter-Finals, Semi-Finals, Finals depending on bracket size)
    - Per-round actions:
      - "Open Voting" button: calls `openMatchupsForVoting` action with all pending matchup IDs in the current round. Disabled if already open or no pending matchups.
      - "Advance Round" button: calls `batchAdvanceRound` action. Enabled only when all matchups in the round have status "decided". Shows count: "Advance 4 matchups".
    - Per-matchup actions (when a matchup is selected):
      - If status is "voting":
        - Vote count display (who's winning)
        - "Close Voting & Accept Result" button: advances the matchup with the vote winner. If tied: shows "Tied! Choose winner or extend voting" with two options
        - "Override: Choose Winner" dropdown: allows teacher to manually select either entrant as winner (calls `advanceMatchup` with chosen entrant)
      - If status is "decided":
        - Winner display
        - "Undo" button: calls `undoAdvancement`. Shows confirmation dialog first. Button is disabled/hidden if next round has votes (the server action enforces this, but UI should also prevent it)
      - If status is "pending":
        - "Open for Voting" button
    - Tie handling: when vote counts are equal, "Close Voting" shows a special UI:
      - "It's a tie! (X - X)"
      - "Break Tie" button with dropdown to manually select winner
      - "Extend Voting" button to keep the matchup open
    - All server action calls wrapped in `useTransition` for non-blocking UI (established pattern from Phase 2)

    Optional countdown timer for voting windows.

      - Color changes: green (>50% time left), yellow (25-50%), red (<25%)
      - "Pause" / "Resume" buttons
      - "Cancel Timer" button
    - When timer expires: calls `onTimerExpire` callback (parent component should auto-close voting or prompt teacher)
    - Timer state saved to bracket via `updateBracketVotingSettings` action (votingTimerSeconds field) so it persists across page refreshes
    - Timer is per-bracket (applies to all open matchups). When voting opens, timer starts. When voting closes, timer stops.
    - If `initialSeconds` is null: show "No timer set" with "Set Timer" button
    - Timer is purely client-side countdown (not server-synced) -- the teacher's browser is the source of truth for the countdown
