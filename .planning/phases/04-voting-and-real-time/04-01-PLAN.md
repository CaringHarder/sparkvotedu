    model Vote {
      id            String             @id @default(uuid())
      matchupId     String             @map("matchup_id")
      matchup       Matchup            @relation(fields: [matchupId], references: [id], onDelete: Cascade)
      participantId String             @map("participant_id")
      participant   StudentParticipant @relation(fields: [participantId], references: [id], onDelete: Cascade)

      entrant       BracketEntrant     @relation(fields: [entrantId], references: [id])
      createdAt     DateTime           @default(now()) @map("created_at")
      updatedAt     DateTime           @updatedAt @map("updated_at")

      @@unique([matchupId, participantId])
      @@index([matchupId])
      @@index([participantId])
      @@map("votes")
    }

<task type="auto">
  <name>Task 2: Vote DAL and server-side Broadcast helper</name>
  <files>src/lib/dal/vote.ts, src/lib/realtime/broadcast.ts</files>
  <action>
    **1. Create src/lib/dal/vote.ts** with these functions:

    - `castVoteDAL(matchupId, participantId, entrantId)`: Uses `prisma.vote.upsert` with `where: { matchupId_participantId: { matchupId, participantId } }` for race-safe idempotent vote creation/update. Returns the upserted vote record.

    - `getVoteCountsForMatchup(matchupId)`: Uses `prisma.vote.groupBy({ by: ['entrantId'], where: { matchupId }, _count: { id: true } })` and reduces into a `Record<string, number>`. Returns `{ [entrantId]: count }`.

    - `hasVoted(matchupId, participantId)`: Uses `prisma.vote.findUnique({ where: { matchupId_participantId: { matchupId, participantId } }, select: { entrantId: true } })`. Returns the vote or null.

    - `getMatchupVoteSummary(matchupId)`: Uses `Promise.all` to fetch total vote count and per-entrant counts in parallel. Returns `{ totalVotes: number, voteCounts: Record<string, number> }`.

    - `getVoterParticipantIds(matchupId)`: Uses `prisma.vote.findMany({ where: { matchupId }, select: { participantId: true } })`. Returns array of participant IDs who have voted -- used for participation grid.

    - `getMatchupStatus(matchupId)`: Fetches matchup status and winnerId. Returns `{ status: string, winnerId: string | null }`.

    - `updateMatchupStatus(matchupId, status)`: Updates the matchup status field. Validates allowed transitions: pending -> voting, voting -> decided. Returns updated matchup.

    - `openMatchupsForVoting(matchupIds)`: Batch update multiple matchups to "voting" status within a transaction. Only updates matchups currently in "pending" status.

    All functions use `import { prisma } from '@/lib/prisma'`. Follow the pattern established in `src/lib/dal/bracket.ts` (no auth checks in DAL -- auth is enforced in server actions).

    **2. Create src/lib/realtime/broadcast.ts** with these functions:

    - `broadcastMessage(message: { topic: string; event: string; payload: Record<string, unknown> })`: Sends a POST to `${SUPABASE_URL}/realtime/v1/api/broadcast` with `apikey: SUPABASE_SERVICE_ROLE_KEY` header and `{ messages: [message] }` body. Uses `process.env.NEXT_PUBLIC_SUPABASE_URL` and `process.env.SUPABASE_SERVICE_ROLE_KEY`. Logs errors to console but does NOT throw (broadcast failure should not break the vote flow).

    - `broadcastVoteUpdate(bracketId, matchupId, voteCounts, totalVotes)`: Calls `broadcastMessage` with topic `bracket:${bracketId}`, event `vote_update`, payload `{ matchupId, voteCounts, totalVotes }`.

    - `broadcastBracketUpdate(bracketId, type, payload)`: Calls `broadcastMessage` with topic `bracket:${bracketId}`, event `bracket_update`, payload `{ type, ...payload }`. The `type` parameter is a union: `'winner_selected' | 'round_advanced' | 'matchup_opened' | 'bracket_completed' | 'voting_opened'`.

    - `broadcastActivityUpdate(sessionId)`: Calls `broadcastMessage` with topic `activities:${sessionId}`, event `activity_update`, payload `{}`. This triggers the existing `useRealtimeActivities` hook to refetch (wiring Phase 2's scaffolded hook).

    **Important:** The broadcast helper must NOT import from Supabase client library. It makes raw HTTP fetch calls using the REST API since server actions are stateless with no persistent WebSocket connection.
  </action>
