  <name>Task 1: Build usePredictionCascade hook</name>
  <files>src/hooks/use-prediction-cascade.ts</files>
  <action>
    Create a new hook file at src/hooks/use-prediction-cascade.ts.

    This hook takes bracket matchups and current selections, and returns:
    - augmentedMatchups: MatchupData[] with speculative entrants filled in from predictions
    - selectableMatchups: MatchupData[] (non-bye matchups where both entrants are known, either real or speculative)
    - selections: Record<string, string> (current state)
    - handleSelect: (matchupId: string, entrantId: string) => void (with cascade logic)
    - totalSelectableCount: number
    - selectedCount: number
    - allSelected: boolean

    Interface:
    ```typescript
    interface UsePredictionCascadeOptions {
      matchups: MatchupData[]
      initialSelections: Record<string, string>
      enabled: boolean  // false when predictions closed or submitted+not editing
    }
    ```

    **Cascade algorithm:**

    The hook maintains two pieces of state:
    - selections: Record<string, string> (matchupId -> predicted winner entrantId)
    - A computed augmentedMatchups derived from selections + original matchups

    **buildAugmentedMatchups(matchups, selections):**
    1. Deep clone the matchups array (map to new objects to avoid mutation).
    2. Build a lookup: matchupById = Map<string, AugmentedMatchup>.
    3. Build a reverse lookup: childrenOf = Map<string, string[]> mapping nextMatchupId -> [matchupId, ...].
       Actually we need the forward lookup: for each matchup M with M.nextMatchupId, M feeds into nextMatchup.
    4. Process selections in round order (sort matchups by round ascending):
       For each matchup M that has a selection (selections[M.id] exists):
       a. Find the selected entrant: the entrant object (entrant1 or entrant2) whose id matches selections[M.id]
       b. Find the next matchup: nextMatchup = matchupById.get(M.nextMatchupId)
       c. If nextMatchup exists, determine which slot M feeds into:
          - If M.position is ODD: fills nextMatchup.entrant1Id / entrant1
          - If M.position is EVEN: fills nextMatchup.entrant2Id / entrant2
          (This matches the existing advancement engine: odd position -> entrant1, even -> entrant2)
       d. Set the speculative entrant on the augmented nextMatchup:
          ```
          if (M.position % 2 === 1) {
            nextMatchup.entrant1Id = selectedEntrant.id
            nextMatchup.entrant1 = { id: selectedEntrant.id, name: selectedEntrant.name, seedPosition: selectedEntrant.seedPosition, bracketId: selectedEntrant.bracketId }
          } else {
            nextMatchup.entrant2Id = selectedEntrant.id
            nextMatchup.entrant2 = { ... }
          }
          ```

    5. Compute selectableMatchups: filter augmentedMatchups where !isBye AND entrant1Id != null AND entrant2Id != null.

    **handleSelect(matchupId, entrantId):**
    1. If !enabled, return early.
    2. Check if the selection changed from a previous value. If selections[matchupId] === entrantId, treat as deselect (remove).
    3. Set selections[matchupId] = entrantId.
    4. **Cascade invalidation:** If the previous selection for this matchup was different:
       a. Find all downstream matchups that were affected by the old selection.
       b. Clear their selections from the selections state.
       c. Algorithm: starting from the current matchup, follow nextMatchupId chain.
          For each downstream matchup D:
          - If selections[D.id] exists AND (D's entrant slot that was populated by the changed matchup now has a different entrant), remove selections[D.id].
          - Continue following D.nextMatchupId.
       This is a simple iterative walk: start at M.nextMatchupId, check if the entrant that was propagated from M has changed, if so clear that downstream selection and continue.

    **Important implementation notes:**
    - Use useMemo for augmentedMatchups computation (depends on [matchups, selections]).
    - Use useCallback for handleSelect.
    - The hook should NOT call any server actions -- it's purely client-side state.
    - selections state initialization should accept initialSelections and sync when they change (same pattern as current predictive-bracket.tsx lines 215-232).
    - Return type should be well-typed, not use 'any'.

    **Type imports:**
    ```typescript
    import type { MatchupData, BracketEntrantData } from '@/lib/bracket/types'
    ```

    Export the hook as a named export: export function usePredictionCascade(...)
