   - Import and call `generateDoubleElimMatchups(effectiveSize)` from double-elim.ts
     (effectiveSize = next power of 2 if using byes, or size if already power of 2)
   - This returns { winners, losers, grandFinals }
   - Create the Bracket record with bracketType='double_elimination'

2. Create entrants as usual (same as single-elim)

3. Create matchups in three passes within the same transaction:
   a. Winners bracket matchups: use createMatchupsInTransaction with bracketRegion='winners'
      - First round matchups get entrant assignments from seed positions
      - Wire nextMatchupId within winners bracket
   b. Losers bracket matchups: create with bracketRegion='losers'
      - All entrant slots are null (filled during advancement when losers drop from WB)
      - Wire nextMatchupId within losers bracket
   c. Grand finals matchup: create with bracketRegion='grand_finals'
      - Both entrant slots null (filled when WB and LB champions determined)
      - No nextMatchupId (this is the final)

   Important: The createMatchupsInTransaction helper needs to accept bracketRegion and set it on each matchup. Extend the helper to accept optional extra fields per matchup (bracketRegion, isBye).

4. If byes exist (non-power-of-two entrants), auto-advance bye matchups in the winners bracket (same pattern as 07-07 for single-elim byes).

5. For play-in enabled brackets (playInEnabled=true on double_elimination), implement the NCAA First Four model:

   a. **Determine play-in entrants:** The play-in adds 8 extra entrant slots beyond the main bracket size. For a 64-team main bracket with play-in enabled, there are 72 total entrants. The lowest 16 seeds (seeds 57-72) play 8 play-in matches. Winners take the 8 lowest seed slots (seeds 57-64) in the main bracket round 1.

   b. **Generate play-in matchups:** Import and call `generatePlayInRound(mainBracketSize, playInCount=8)` from double-elim.ts (or the play-in generation function from RESEARCH.md). This returns:
      - `playInMatchups`: Array of MatchupSeed objects with round=0
      - `mainBracketSlotsToFill`: Which main bracket seed positions are filled by play-in winners

      The play-in matchup pairing logic:
      - Match 1: seed (mainSize - playInMatchCount + 1) vs seed (totalEntrants) -- e.g., seed 61 vs seed 72
      - Match 2: seed (mainSize - playInMatchCount + 2) vs seed (totalEntrants - 1) -- e.g., seed 62 vs seed 71
      - Continue for all playInMatchCount matches (8 extra entrants = 4 matches)
      - Each play-in matchup pairs the highest available play-in seed against the lowest

   c. **Create play-in matchups in the database:** Within the same transaction, create play-in matchups with:
      - `bracketRegion = 'winners'` (they feed into the winners bracket)
      - `round = 0` (before the main bracket round 1)
      - `position = 1..playInMatchCount`
      - Both entrant slots filled with the actual play-in entrants (these are real matches, not byes)
      - `isBye = false`

   d. **Wire play-in winners into main bracket:** Set `nextMatchupId` on each play-in matchup to point to the correct round 1 WB matchup. The target matchup is whichever R1 matchup has the corresponding seed slot:
      - Play-in match for seeds 61 vs 72: winner fills the seed-61 slot in the R1 matchup that would normally have seed 61
      - Use the existing seeding algorithm (`buildSeedOrder`) to find which R1 matchup position corresponds to each seed

   e. **Do NOT auto-advance play-in matchups.** Unlike byes, play-in matches require actual competition (voting). Both entrants are real.

   f. **Leave the main bracket R1 entrant slots empty** for positions that will be filled by play-in winners. These slots get filled when the play-in matchup is advanced.

6. Return the full bracket with all matchups including bracket region information.

Key implementation note: The createMatchupsInTransaction helper is currently a local function in bracket.ts. To support bracketRegion, extend it to accept an optional `extraFields: Record<string, unknown>` parameter that gets spread into each matchup.create call. This is cleaner than making it accept every possible field individually.

  <name>Task 2: Double-elimination advancement with loser drop and grand finals</name>
  <files>src/lib/bracket/advancement.ts, src/actions/bracket-advance.ts</files>
  <action>
1. Add advanceDoubleElimMatchup function to advancement.ts:

   This function wraps the existing advanceMatchupWinner with double-elim-specific logic:

   a. Call advanceMatchupWinner normally (sets winner, propagates to next matchup in same region)
   b. ADDITIONALLY, if the matchup is in the winners bracket (bracketRegion='winners'):
      - The LOSER must drop to the losers bracket
      - Determine which LB matchup to drop into:
        - Use the seedLosersFromWinnersRound function from double-elim.ts to determine the correct LB position
        - WB round R losers drop into LB major round (the round that accepts WB dropdowns)
        - The specific LB slot depends on the WB matchup position and the split-and-reverse seeding
      - Place the loser into the correct entrant slot of the target LB matchup

   c. If the matchup is the final matchup in losers bracket (LB champion):
      - Place the LB champion into grand finals matchup entrant2 slot

   d. If the matchup is the final matchup in winners bracket (WB champion):
      - Place the WB champion into grand finals matchup entrant1 slot

   e. If the matchup is grand finals match 1:
      - If WB champion wins: bracket is complete (no reset needed). Set bracket status to 'completed'.
      - If LB champion wins: create a RESET matchup dynamically:
        - Create a new matchup with bracketRegion='grand_finals', round=2, position=1
        - Set entrant1=WB champion, entrant2=LB champion
        - This is the "if necessary" match. It was NOT created upfront (per CONTEXT.md decision).

   f. If the matchup is grand finals match 2 (reset): winner is the bracket champion. Set bracket status to 'completed'.

   All mutations in a single $transaction.

2. Add server actions to bracket-advance.ts:
   - advanceDoubleElimMatchup action: auth -> validate -> call advanceDoubleElimMatchup -> broadcast -> revalidate
   - This action is distinct from the existing advanceMatchup action to avoid cluttering single-elim logic
   - The action checks bracketType and routes to the correct advancement function

   Alternative approach (preferred): Update the existing advanceMatchup action to detect bracketType and call the appropriate advancement function. This keeps a single action endpoint:
   ```typescript
   if (bracket.bracketType === 'double_elimination') {
     result = await advanceDoubleElimMatchup(matchupId, winnerId, bracketId)
   } else {
     result = await advanceMatchupWinner(matchupId, winnerId, bracketId)
   }
   ```

3. Add a helper to find the correct LB target matchup for a WB loser:
   This needs to:
   - Know which WB round the loser came from
   - Map WB round R to the LB major round that accepts WB R losers
   - Map WB matchup position to LB matchup position using split-and-reverse
   - Query the DB for the target LB matchup by bracketId + bracketRegion='losers' + round + position
