**validation.ts:** Add `reopenBracketSchema` Zod schema with a single `bracketId` field (uuid string). Add `reopenPollSchema` Zod schema with a single `pollId` field (uuid string). Export both schemas and their inferred types (`ReopenBracketInput`, `ReopenPollInput`).

**bracket.ts (DAL):** Add `reopenBracketDAL(bracketId: string, teacherId: string)` function. This function:
1. Finds the bracket with `{ id: bracketId, teacherId, status: 'completed' }` selecting `id`, `bracketType`, `sessionId`
2. Returns `{ error: 'Bracket not found or not completed' }` if not found
3. Calls `getMostRecentAdvancedRound(bracketId, bracket.bracketType)` from `advancement.ts` to determine the final round
4. Returns `{ error: 'No advanced rounds found' }` if finalRound is null
5. Dispatches to the appropriate undo function based on `bracketType`:
   - `single_elimination` -> `undoRoundSE(bracketId, finalRound.round)`
   - `round_robin` -> `undoRoundRR(bracketId, finalRound.round)`
   - `double_elimination` -> `undoRoundDE(bracketId, finalRound.round, finalRound.region!)`
   - `predictive` -> `undoRoundPredictive(bracketId, finalRound.round)`
6. After undo, transitions status `completed -> paused` via direct prisma update (bypasses VALID_TRANSITIONS, same pattern as `unarchiveBracketDAL`)
7. Returns `{ success: true, sessionId: bracket.sessionId }`

Import `getMostRecentAdvancedRound`, `undoRoundSE`, `undoRoundRR`, `undoRoundDE`, `undoRoundPredictive` from `@/lib/bracket/advancement`.

Note: Champion designation is derived data — it is read from the final matchup's `winnerId`, NOT stored as a separate database column. Calling the undo engine (e.g., `undoRoundSE`) clears the final matchup's `winnerId`, which implicitly clears the champion. No additional champion-clearing logic is needed.

**poll.ts (DAL):** Add `reopenPollDAL(pollId: string, teacherId: string)` function. This function:
1. Finds the poll with `{ id: pollId, teacherId, status: 'closed' }` selecting `id`, `sessionId`
2. Returns `{ error: 'Poll not found or not closed' }` if not found
3. Transitions status `closed -> paused` via direct prisma update (bypasses VALID_POLL_TRANSITIONS, same pattern as unarchive)
4. Sets `allowVoteChange: false` as part of the update to enforce "already-voted students locked out" per user decision
5. Returns the updated poll object

Note: Do NOT modify VALID_TRANSITIONS or VALID_POLL_TRANSITIONS. The `completed -> paused` and `closed -> paused` transitions use dedicated DAL functions with direct prisma updates, following the established pattern for reverse operations.

**bracket-advance.ts:** Add `reopenBracket` server action (exported, `'use server'`). Pattern follows `undoRoundAdvancement`:
1. `getAuthenticatedTeacher()` for auth
2. Validate input with `reopenBracketSchema` (only bracketId)
3. Call `reopenBracketDAL(bracketId, teacher.id)`
4. If error, return `{ error }`
5. Broadcast `broadcastBracketUpdate(bracketId, 'bracket_reopened', {})` (fire-and-forget with `.catch(console.error)`)
6. Also broadcast on the activity channel if sessionId exists: `broadcastBracketUpdate(bracketId, 'bracket_reopened', {})` -- the existing broadcast function handles this
7. Call `revalidatePath` for the bracket detail and live pages
8. Return `{ success: true }`

**poll.ts:** Add `reopenPoll` server action (exported, `'use server'`). Pattern follows `updatePollStatus`:
1. `getAuthenticatedTeacher()` for auth
2. Validate input with `reopenPollSchema` (only pollId)
3. Call `reopenPollDAL(pollId, teacher.id)`
4. If error, return `{ error }`
5. Broadcast `broadcastPollUpdate(pollId, 'poll_reopened')` (fire-and-forget with `.catch(console.error)`)
6. Call `revalidatePath` for the poll detail and live pages

**use-realtime-bracket.ts:** In the `fetchBracketState` callback, after the existing `if (data.status === 'completed') { setBracketCompleted(true) }` block, add an `else` branch: `else { setBracketCompleted(false) }`. This ensures that when a completed bracket is reopened and refetched, the `bracketCompleted` flag is reset to false, dismissing the celebration screen. Also add `bracket_reopened` to the list of event types that trigger a refetch (find the event handler where `bracket_paused`, `bracket_resumed`, `round_undone` are listed and add `bracket_reopened`).

**use-realtime-poll.ts:** Add `poll_reopened` to the list of event types that trigger a full refetch (find where `poll_paused`, `poll_resumed` are listed and add `poll_reopened`).

**simple-voting-view.tsx:** Add a `useEffect` that watches `bracketCompleted` and resets celebration state when it becomes false (i.e., when the bracket is reopened). This is necessary because `showCelebration` is latched to `true` by a timer after `bracketCompleted` becomes true, and never resets on its own:
```
useEffect(() => {
  if (!bracketCompleted) {
    setShowCelebration(false)
    setRevealState(null)
  }
}, [bracketCompleted])
```
Place this effect immediately after the existing `bracketCompleted` celebration timer effect.

**advanced-voting-view.tsx:** Add an equivalent `useEffect` that watches `bracketCompleted` and resets all celebration/reveal state when it becomes false:

  }

Place this effect immediately after the existing fallback WinnerReveal effect. Resetting `hasShownRevealRef` is necessary so that the reveal can trigger again if the bracket is later re-completed.
